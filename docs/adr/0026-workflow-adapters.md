# ADR 0026: Workflow Adapters — Restate and DBOS

- **Status:** Accepted (Phase 2a — Restate runtime; Phase 2b — DBOS; Phase 2c — codegen)
- **Date:** 2026-05-14
- **Pairs with:** ADR 0012 (event delivery), ADR 0025 (workflow-orchestrated command bus).

## Context

ADR 0025 landed the `cmdworkflow` package with a narrow
`WorkflowRuntime` interface (`Run` + `Spawn`) and an `inproc`
adapter for tests. Production deployments need durability: workflows
that survive process restarts, retries that are journaled, dedup
that is enforced cross-call. Two real runtimes are available:

- **Restate** — durable invocation runtime, separate process,
  Go SDK at `github.com/restatedev/sdk-go` v0.24.0+. Mature, polyglot
  (Go / TS / Java / Kotlin / Python).
- **DBOS** — Postgres-native workflow journal, Go SDK now at parity
  with TS/Python. Co-locates workflow state with our eventstore
  tables — one DB, one backup story.

This ADR captures how both adapters bridge into our
`WorkflowRuntime` shape, how per-aggregate services are exposed, and
how command codegen generates typed handlers driven by a proto
annotation.

## Decision

### 1. Context bridging via `context.Value` (Option A)

`restate.Context` is an interface that embeds the stdlib
`context.Context`. We do NOT widen our `WorkflowRuntime` signatures
to take a runtime-specific context type — that would force every
adapter (including `inproc`) to deal in a transport handle even when
they don't need one.

Instead, the adapter stashes the runtime-specific context inside a
stdlib `context.Context` via a typed key, extracts it in `Run` and
`Spawn`. The bridge is one wrapping call at the entry of every
runtime-specific handler:

```go
// User-written Restate handler — one wrapping call:
func (s *InvoiceService) Create(ctx restate.Context, cmd *invoicev1.Create) (*invoicev1.Invoice, error) {
    stdCtx := cwrestate.WithContext(context.Background(), ctx)
    return s.workflow.HandleCmd(stdCtx, deriveStreamID(cmd), cmd)
}
```

Inside `cwrestate.Runtime.Run`, the adapter extracts `restate.Context`
from the stdlib value, calls `restate.Run(rc, fn)`, and re-wraps the
nested `RunContext` for any callbacks. Same shape for DBOS:
`cwdbos.WithContext(ctx, dbosCtx)`.

The unexported `inner()` method on `restate.Context` means
applications cannot implement the interface themselves outside the
SDK — so we type-assert against the concrete SDK interface, never
mock it directly.

### 2. Per-aggregate Restate service (Option B confirmed for Phase 2c)

Restate organizes durable handlers as **services**. We codegen one
service per aggregate, with one handler per command type:

```go
// Generated by protoc-gen-es-go for the Invoice aggregate.
type InvoiceService struct {
    workflow *cmdworkflow.Workflow[*invoicev1.Invoice, invoicev1.Command]
}

func (s *InvoiceService) Create(ctx restate.Context, cmd *invoicev1.Create) (*invoicev1.Invoice, error) { ... }
func (s *InvoiceService) MarkPaid(ctx restate.Context, cmd *invoicev1.MarkPaid) (*invoicev1.Invoice, error) { ... }
func (s *InvoiceService) Void(ctx restate.Context, cmd *invoicev1.Void) (*invoicev1.Invoice, error) { ... }
```

The user registers once with Restate:

```go
srv := server.NewRestate().Bind(restate.Reflect(invoiceService))
srv.Start(ctx, ":9080")
```

Restate's HTTP ingress then routes `POST /InvoiceService/Create` to
the handler, which calls into the framework's `Workflow.HandleCmd`.

Per-command-type handlers (vs one generic `HandleCmd(any) any`):

- Restate's reflection-based binding gives each handler a stable URL
  for ingress (`/Service/method`), with method-level visibility,
  retries, retention. A single generic handler would lose this.
- Typed inputs/outputs — Restate's codec (proto or protojson) does
  serialization with the right schema. The framework never round-
  trips through `any`.
- Per-command idempotency, per-command observability dashboards.

### 3. Stream ID from a proto annotation

The codegen needs to know which command field provides the
`StreamID`. We add a proto field annotation:

```protobuf
import "eventstore/v1/annotations.proto";

message Create {
  string invoice_id = 1 [(eventstore.v1.stream_id) = true];
  string customer_id = 2;
  ...
}

message MarkPaid {
  string invoice_id = 1 [(eventstore.v1.stream_id) = true];
  ...
}
```

Codegen detects the annotated field and emits:

```go
func deriveStreamID(cmd invoicev1.Command) es.StreamID {
    switch c := cmd.(type) {
    case *invoicev1.Create:   return es.NewStreamID(tenant, "invoice", c.InvoiceId)
    case *invoicev1.MarkPaid: return es.NewStreamID(tenant, "invoice", c.InvoiceId)
    case *invoicev1.Void:     return es.NewStreamID(tenant, "invoice", c.InvoiceId)
    }
}
```

The aggregate's stream type (`"invoice"`) comes from the proto
package or an explicit `[(eventstore.v1.stream_type) = "invoice"]`
on the parent file. Tenant comes from the call context (extracted
from a header in the Restate handler).

**Escape hatch**: commands that derive stream id from multiple
fields or from a different shape annotate with
`[(eventstore.v1.stream_id_via) = "GetDerivedID"]` and the user
hand-writes the `GetDerivedID() string` method. Codegen falls back
to calling it.

**Default**: if no annotation is present on any field, codegen
emits a handler signature with `streamID` as a separate parameter
(Option B from the design discussion). Lets users opt out of the
convention without breaking codegen.

### 4. DBOS shares Postgres with the eventstore

DBOS's workflow journal is a set of tables in Postgres. We do NOT
require a separate database — DBOS lives in the same PG instance as
the eventstore by default, in its own schema (`dbos`). Backups,
migrations, and consistency boundaries unify.

The DBOS adapter's `Runtime.Run` uses the DBOS Go SDK's `Step`
primitive (the journaled-step equivalent of Restate's `Run`).
`Spawn` uses DBOS's `StartChildWorkflow`. Idempotency key flows
through DBOS's `IdempotencyKey` workflow option.

When SQLite is the eventstore: DBOS requires Postgres for its
journal, so SQLite eventstore + DBOS workflows is **not a supported
combination**. Use the inproc adapter for SQLite + workflow tests,
or run DBOS against a separate PG. Documented in the cookbook.

### 5. Test scope (build-tagged, container-backed)

Tests for the Restate adapter are tagged `restate` and use the SDK's
`testing.Start` helper which manages a `restatedev/restate:latest`
container plus an in-test SDK server. Four scenarios verify the
behaviors that matter most:

- **Durability under restart** — start a flow, kill the SDK process
  mid-Run, restart, verify journal replay completes the workflow
  exactly once at the eventstore boundary.
- **Invocation-key dedup** — call `Create` twice with the same
  `idempotency-key` header, verify the second returns the cached
  result without re-Appending.
- **Compensation roundtrip** — `Sync + Compensate` subscriber
  exhausts, the compensating command appends as a nested workflow
  invocation, the journal shows both.
- **Async fan-out durability** — kill mid-Spawn'd-child, verify the
  child resumes from its own journal after restart.

DBOS tests follow the same shape against a Postgres testcontainer
(already in CI). Build-tagged `dbos`.

Load tests are out of scope for Phase 2 — they fit in
loadtest-tagged files like the existing eventstore load tests.

### 5b. Parallel Sync via `restate.RunAsync`

Per ADR 0025 § 4 (revised), Sync subscribers fan out in parallel via
`RunAsync`. The Restate adapter maps `WorkflowRuntime.RunAsync` to
the SDK's `restate.RunAsync[[]byte](rc, fn, WithName(name))`, which
returns a `RunAsyncFuture[[]byte]`. Multiple `RunAsync` calls from
one handler invocation execute concurrently in Restate's runtime;
the framework awaits all futures before proceeding.

Each `RunAsync` is one journal entry. Retries happen inside the
fn (not as nested `restate.Run` calls — the SDK's RunContext doesn't
allow it). One subscriber × one event × N retry attempts = exactly
one journal entry.

DBOS adapter does the analogous mapping against DBOS's `StartChildStep`
primitive.

### 5c. OnExhausted policy runs from the outer HandleCmd context

Two related concerns made the naive "policy inside RunAsync's fn"
shape impossible for Restate:

1. The SDK's `RunContext` lacks `inner()`; any nested `restate.Run`
   call from inside a `RunAsync` closure fails with
   `ErrNoRestateContext`.
2. A non-nil error return from `RunAsync`'s fn is treated by Restate
   as step failure → whole-invocation retry. Using fn-error as the
   "exhausted" signal triggers infinite retry loops on any failed
   subscriber.

Fix: `runSyncSubscriber`'s fn ALWAYS returns nil error. The
exhausted-with-lastErr signal travels through the `[]byte` return
(non-empty = exhausted, message = lastErr). `HandleCmd`'s outer
loop decodes this and applies OnExhausted policy from the
top-level context where the real `restate.Context` is in scope.

DLQ insert + Compensate recursion both run as fresh `Run` calls
issued from `HandleCmd` — not nested inside any closure.

Compensate recursion needs unique step names to avoid journal
collisions with the parent's "append" / "read-envelopes". The
framework pushes a `stepPrefix` value on context for nested
HandleCmd calls; all step names in `appendStep`,
`readEnvelopesStep`, `runSyncSubscriber`, and `onExhausted` honor
the prefix. Compensate uses `"compensate:<sub>:<event>:"` —
deterministic across replays.

All four policy combinations (`Drop` / `DLQ` / `Compensate` × `Sync`
/ `Async`) are operational on the Restate adapter.

### 6. Payload encoding — `BinaryCodec`

Restate supports JSON / proto / protojson / binary codecs. The
framework uses **`BinaryCodec`** for adapter-internal payloads: the
gob-encoded `wireAppendResult` and `[]wireEnvelope` we already use
in `cmdworkflow/codec.go` flow through unchanged. The application's
own command and state types use **`ProtoCodec`** (proto binary) for
Restate's typed handler signatures.

Two distinct codec choices:

- **Framework internals** (step results): `BinaryCodec`, opaque bytes.
- **User-facing types** (commands, return states): `ProtoCodec`.

### 6b. Durable Async fan-out via per-subscriber workflows

Phase 2a wired Async subscribers as in-process goroutines — fire and
forget, not journaled. A process crash mid-Async lost in-flight
work. This is replaced with durable child workflows:

- Each (Async subscriber × event) becomes one runtime workflow
  invocation with the deterministic id
  `"<streamType>:<subscriberName>:<eventID>"` — globally unique
  (eventIDs are UUIDv7), retry-safe across replays.
- Codegen emits an `AsyncDispatch(ctx, payload AsyncPayload)` method
  on each Service struct (RestateService + DBOSService).
- The Service constructor wires
  `wf.SetAsyncSend(s.sendAsync)` — the framework's spawnAsync
  delegates to it.
- `sendAsync` invokes the runtime's send primitive:
  - Restate: `restate.ServiceSend(rc, "Invoice", "AsyncDispatch").Send(payload, WithIdempotencyKey(workflowID))`
  - DBOS: `dbos.RunWorkflow(dctx, s.AsyncDispatch, payload, WithWorkflowID(workflowID))` with handle discarded

The dispatched workflow runs the full subscriber retry + OnExhausted
policy under its own durable journal. Failures result in DLQ inserts
(also journaled steps within the child workflow).

`Async + Compensate` is now panic-on-register at framework level —
saga compensation only makes sense for Sync subscribers where the
original caller is waiting. Fire-and-forget compensation has no
recovery story for the caller. Use Sync+Compensate for saga steps;
DLQ or Drop for Async failures.

Inproc adapter continues to use a goroutine for Spawn (not durable
— inproc is test-only anyway).

### 7. Saga primitives (`Sleep`, `Wait`, `Awakeable`, `Cancel`) deferred

This ADR does not extend `WorkflowRuntime` with the saga primitives
named in ADR 0025's deferral list. The current two-method interface
(`Run` + `Spawn`) is sufficient for Sync, Async, and Compensate
subscribers. Long-running waits, timers, and external-event
awakeables come in a future ADR alongside the saga API surface;
that ADR will widen `WorkflowRuntime` with the new methods and both
adapters add implementations.

## Consequences

### Positive

- **Production deployable** for the workflow-orchestrated command
  bus on both Restate and DBOS.
- **Codegen ergonomics** — one method per command type with typed
  inputs/outputs and stable HTTP routes.
- **Idempotency is real** at the framework level (was theoretical
  with inproc).
- **DBOS unifies storage** — one Postgres instance for eventstore +
  workflow journal.

### Negative

- **Restate is a separate runtime to operate.** Container, network,
  upgrades, observability — non-trivial for small teams.
- **DBOS-Go SDK is younger than Restate's**. Public API stability is
  good but smaller user base; bug surface is real.
- **Codegen pass complexity grows.** The plugin already emits codec,
  command/event sum types, and projection scaffolding. Adding
  service definitions per aggregate is mechanical but increases the
  amount of generated code teams have to scan during review.
- **SQLite + DBOS is unsupported.** Application teams using SQLite
  eventstore for tests get the inproc workflow adapter, not DBOS.
  Documented; not a real limitation since SQLite is the dev/test
  target, not the production path.

### Neutral

- **Two adapters maintained in parallel.** Same shape, different
  dependencies. Conformance suite will live in `cmdworkflow/cwtest`
  so any future adapter (Temporal? Step Functions?) drops in
  against the same tests.

## Alternatives Considered

### Single Restate handler taking a generic command interface

Rejected for Phase 2c. Loses typed input/output, per-command HTTP
routes, and per-command observability. The codegen cost is low
enough that per-command-type handlers win clearly.

### Embedding restate.Context as a field on Workflow

Rejected. Couples the framework type to the Restate SDK. Stdlib
`context.Context` plumbing keeps the framework agnostic.

### Polling-based workflow runtime (a third adapter)

Rejected. ADR 0001 already covers polling-based deployment via
in-process projection.Runtime. The whole motivation for
`cmdworkflow` is workflow-runtime-driven delivery; a polling
"runtime adapter" would defeat the design.

### Both adapters share a generated service struct

Rejected as premature. Codegen emits one struct per (aggregate ×
adapter) for now. If patterns converge, a future refactor can
extract a shared base. The codegen plugin is single-source-of-truth
for the shape so generated code stays consistent across adapters
without sharing types.
