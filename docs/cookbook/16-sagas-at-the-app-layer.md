# 16: Sagas at the App Layer

Saga orchestration — multi-step processes with timers, external
callbacks, concurrent waits — **lives in your application code, not
the framework**. The eventstore provides command bus + subscribers
(the "what changed"); the workflow runtime (Restate, DBOS) provides
`Sleep`, `Awakeable`, `Select`; your app composes them.

This recipe shows the pattern with a worked Restate example. The DBOS
version is mechanically the same — substitute `dbos.Sleep`,
`dbos.RunWorkflow`, etc. Both worked examples ship in the repo:

- Restate: `examples/cmdworkflow-restate/`
- DBOS: `examples/cmdworkflow-dbos/` — including the
  **Sync+Compensate saga step** under DBOS (see
  `TestDBOSExample_SagaCompensation`), which proves the
  compensation contract holds end-to-end through the codegen-emitted
  `DBOSService`.

## Why not in the framework?

Two reasons. **Scope discipline**: a state-management framework that
absorbs orchestration primitives bloats indefinitely (every new saga
pattern → framework PR). **Compositional clarity**: a saga is a Go
function that calls the framework — there's no abstraction that
sits between them.

ADR 0026 § 7 makes the boundary explicit.

## The shape

A saga is a Restate / DBOS handler:

```go
package myapp

import (
    "context"

    restate "github.com/restatedev/sdk-go"
    cwrestate "github.com/laenenai/eventstore/adapters/cmdworkflow/restate"
    invoicev1restate "github.com/laenenai/eventstore/adapters/cmdworkflow/restate/gen/myapp/invoice/v1"
    invoicev1 "github.com/laenenai/eventstore/gen/myapp/invoice/v1"
)

type OrderSaga struct {
    InvoiceSvc *invoicev1restate.RestateService  // generated by codegen
    Gateway    PaymentGateway                    // your external dep
}

// CompleteRefund is a Restate workflow — the saga.
// Bound at startup alongside the framework's per-command handlers.
func (s *OrderSaga) CompleteRefund(rc restate.Context, req RefundRequest) (RefundResult, error) {
    stdCtx := cwrestate.WithContext(context.Background(), rc)

    // Step 1: call the payment gateway, wait for its webhook.
    awakeable := restate.NewAwakeable[*PaymentSettled](rc)
    if err := s.Gateway.RequestRefund(rc, req.OrderID, req.Amount, awakeable.ID); err != nil {
        return RefundResult{}, err
    }
    payment, err := awakeable.Result()  // suspends DURABLY — minutes, hours, days
    if err != nil {
        return RefundResult{}, err
    }

    // Step 2: 7-day customer-grace window.
    restate.Sleep(rc, 7*24*time.Hour)  // durable timer, survives restarts

    // Step 3: invoke the framework's command bus to record the refund.
    state, err := s.InvoiceSvc.MarkRefunded(rc, &invoicev1.MarkRefunded{
        TenantId:  req.TenantID,
        InvoiceId: req.OrderID,
        Amount:    payment.Amount,
        Ref:       payment.GatewayRef,
        SettledAtMs: time.Now().UnixMilli(),
    })
    if err != nil {
        return RefundResult{}, err
    }
    return RefundResult{Status: state.Status, FinalAmount: state.TotalCents}, nil
}
```

## Wiring at startup

```go
// main.go
saga := &myapp.OrderSaga{InvoiceSvc: invoiceSvc, Gateway: gateway}
srv := server.NewRestate().
    Bind(restate.Reflect(invoiceSvc)).            // framework handlers
    Bind(restate.Reflect(saga))                   // app saga handlers
srv.Start(ctx, ":9080")
```

Restate routes incoming `/OrderSaga/CompleteRefund` calls to the
saga. The framework's `/Invoice/MarkRefunded` still routes through
the generated `RestateService`. Both are first-class handlers.

## Invoking the saga

From your Connect-go HTTP layer:

```go
client := ingress.Service[RefundRequest, RefundResult](
    restateClient, "OrderSaga", "CompleteRefund")
result, err := client.Request(ctx, req,
    restate.WithIdempotencyKey(req.RequestID))
```

The HTTP request returns the saga's final result. If the caller
disconnects mid-saga (between Steps 1 and 2), Restate keeps the
journal — invoke `Attach` later to fetch the eventual result.

## DBOS variant

Same shape, DBOS primitives:

```go
type OrderSaga struct {
    InvoiceSvc *invoicev1dbos.DBOSService
    Gateway    PaymentGateway
}

func (s *OrderSaga) CompleteRefund(dctx dbos.DBOSContext, req RefundRequest) (RefundResult, error) {
    // Step 1: send refund request, suspend on awakeable.
    notification, _ := dbos.GetEvent[*PaymentSettled](dctx, "", "settled-"+req.OrderID, 0)
    s.Gateway.RequestRefund(dctx, req.OrderID, req.Amount, "settled-"+req.OrderID)
    payment := <-notification

    // Step 2: durable Sleep.
    dctx.Sleep(dctx, 7*24*time.Hour)

    // Step 3: framework command via the codegen-emitted method.
    return s.InvoiceSvc.MarkRefunded(dctx, &invoicev1.MarkRefunded{...})
}
```

(DBOS uses `Send/GetEvent` for the callback pattern; Restate uses
`Awakeable`. Same shape, different primitives.)

### Inline compensation under DBOS

For the narrower case where a single subscriber's exhaustion should
roll back the command (the framework's `Sync + Compensate` knob — not
a saga, but adjacent), the DBOS example exercises the worked path:

- Subscriber wiring: `examples/cmdworkflow-dbos/subscribers.go`,
  `CreditReservation` — `Mode: Sync`, `OnExhausted: Compensate`,
  `Compensate` returns the inverse command (e.g., `Void`).
- End-to-end test: `examples/cmdworkflow-dbos/example_test.go`,
  `TestDBOSExample_SagaCompensation` — declines deterministically,
  asserts `Created + Voided` in the stream, asserts the active-invoices
  read model dropped the voided row.

The compensating recursion runs under the same `DBOSContext` as the
original `HandleCmd` with step names prefixed
`compensate:<sub>:<event>:` — DBOS journals it distinctly from the
primary handler, so a crash mid-compensation resumes from the journal.

## What this gives you

- **One function per saga**. The whole flow reads top-to-bottom.
- **Durable across restarts**. Process crashes mid-Sleep / mid-Awakeable
  resume from the journal.
- **Idempotent.** Restate / DBOS dedup the saga invocation by
  workflow id (idempotency key); replays of any step return the
  cached result.
- **The eventstore is unaware**. From the framework's perspective,
  `MarkRefunded` is just a command arriving — same as if a UI button
  caused it. Subscribers fire normally.

## What stays in the framework

These framework primitives serve sagas without being saga-shaped
themselves:

| Need | Framework primitive |
| ---- | ------------------- |
| Record state changes durably | `bus.HandleCmd` |
| Read-your-writes view of state | Tier 1 `state_cache` |
| Project events into a read model | `projection.Runtime` |
| Mirror current state externally | `state_stream.Drain` |
| Roll back inline (single-step) | `Sync + Compensate` Subscriber |

The framework's `Sync + Compensate` (ADR 0025) is **not** a saga —
it's an in-line transactional rollback for one subscriber's
exhaustion. Sagas span multiple commands, multiple aggregates, and
typically time.

## Anti-patterns

- **Don't try to model sagas as long-lived Decider state.** Aggregates
  represent things (orders, invoices); sagas represent processes.
  Aggregates respond to commands one at a time; sagas drive command
  sequences.

- **Don't make Subscribers wait.** A Subscriber's `Handle` should be
  millisecond-scale (UPSERT, API call, log write). If you find
  yourself wanting `time.Sleep(30 * time.Minute)` inside a Handle,
  you've got a saga — move it out.

- **Don't put sagas inside `cmdworkflow.Workflow.HandleCmd`.**
  HandleCmd is request-scoped. Sagas may span hours or days. They
  live at the same level as HandleCmd — both are workflow handlers
  registered with the runtime.

## Testing sagas

Use the runtime's testing harness:

- Restate: `adapters/cmdworkflow/restate/testsupport` spawns a
  container. Invoke your saga via the ingress client; verify the
  framework's eventstore state.
- DBOS: `adapters/cmdworkflow/dbos/testsupport` spawns Postgres.
  Invoke via `dbos.RunWorkflow(dctx, saga.CompleteRefund, req)`.

For unit tests that don't need durability, the inproc workflow
runtime works for the framework side, and you mock the runtime's
saga primitives at the boundary.
